Part 1 · Everything Bitcoin taught us

History

“Satoshi's blockchain was the first credible decentralized solution.”

What this part is doing: before proposing anything, the paper spends a third of its length re-deriving Bitcoin — not as a history lesson, but as a setup. Every Bitcoin concept it explains (state transitions, mining, Merkle trees, scripting) is chosen because Ethereum will either generalize it or fix it. Decades of digital cash attempts (Chaum's e-cash, Wei Dai's b-money, Hal Finney's reusable proofs of work) all failed on the same rock: decentralized consensus. Academic Byzantine-fault-tolerance research assumed you knew who the participants were; on the open internet an attacker can invent a million fake identities (a sybil attack). Satoshi's fix was to make voting power proportional to computing power — costly to fake — and it worked well enough to carry hundreds of currencies within five years.

1.1Bitcoin as a State Transition System

“From a technical standpoint, the ledger of a cryptocurrency such as Bitcoin can be thought of as a state transition system.”

This is the paper's master abstraction, and everything else hangs off it. Any ledger — a bank, Bitcoin, Ethereum — is just a state (who has what) plus a rule APPLY(S, TX) → S′ or ERROR that takes the current state and a transaction and produces the next state, rejecting anything invalid. In a bank, the state is a balance sheet. In Bitcoin, the state is the set of unspent coins (UTXO), and a transaction consumes some coins and mints new ones, conserving value and checking signatures.

Why frame it this way? Because once you see Bitcoin as “a consensus system that agrees on the order in which a state transition function was applied,” an obvious question appears: who says the state transition function has to be about coins? The whole Ethereum pitch is: keep the consensus machinery, make APPLY programmable.

code · APPLY, the paper's first formalism
// A bank as a state transition system — the paper's exact example.
// State: who has what.  Transaction: a requested change.
function APPLY(S, tx){
  const S2 = { ...S };                       // never mutate history
  if ((S2[tx.from] || 0) < tx.amount) return 'ERROR: insufficient funds';
  S2[tx.from] -= tx.amount;                  // debit the sender…
  S2[tx.to] = (S2[tx.to] || 0) + tx.amount;  // …credit the recipient
  return S2;
}

const S = { Alice: 50, Bob: 50 };
print('APPLY(S, "send $20 Alice→Bob") = ' + JSON.stringify(APPLY(S, {from:'Alice', to:'Bob', amount:20})));
print('APPLY(S, "send $70 Alice→Bob") = ' + JSON.stringify(APPLY(S, {from:'Alice', to:'Bob', amount:70})));
// Bitcoin hard-codes one APPLY about coins. Ethereum's entire thesis:
// let users upload their own APPLY — that upload is a "smart contract".

1.2Mining

“The purpose of this is to make block creation computationally ‘hard’, thereby preventing sybil attackers from remaking the entire blockchain in their favor.”

With no central server, everyone must agree on the order transactions were applied — order is everything when the same coin is spent twice. Bitcoin's answer: miners race to package transactions into blocks, each block must contain a proof-of-work (a hash below a target, findable only by brute force), and in a fork the chain backed by the most work wins. An attacker who wants to un-spend a payment must out-compute the entire honest network — the "51% attack" threshold. Ethereum keeps this consensus engine essentially intact; the paper's quarrel is never with Bitcoin's consensus, only with what's inside the blocks.

1.3Merkle Trees

“The purpose of the Merkle tree is to allow the data in a block to be delivered piecemeal.”

A block's hash doesn't cover its transactions directly — they're hashed pairwise in layers into a tree, and only the root enters the header. Because hashes propagate upward, tampering with any leaf changes the root; and proving one transaction is in a block needs only the handful of hashes along its branch, not the whole block. This is what lets phones verify payments ("light nodes" / SPV) while full history lives elsewhere. Hold this thought — Ethereum is about to bet the entire world state, not just transaction lists, on a souped-up version of this structure (§2.5).

1.4Alternative Blockchain Applications

“There are two approaches toward building a consensus protocol: building an independent network, and building a protocol on top of Bitcoin.”

By 2014 people were already using blockchains for more than money: Namecoin (first-come-first-served name registration — decentralized DNS), colored coins (tag a bitcoin to represent a stock or a dollar), metacoins (whole new protocols piggybacking their data inside Bitcoin transactions). The paper's diagnosis: both existing strategies are bad. Launching your own blockchain means re-bootstrapping mining, networking, and security from zero for every app — and most apps are too small to deserve it. Piggybacking on Bitcoin is easier but broken for light clients: Bitcoin's miners don't validate your meta-protocol's rules, so a phone can't trust chain depth as proof of validity and must trust a server instead — exactly the thing cryptocurrency exists to avoid.

Key idea: apps need consensus as a shared service, the way websites need computation as a service. What's missing is the platform — a base layer that provides security once, for every application at the same time.

1.5Scripting

“Even without any extensions, the Bitcoin protocol actually does facilitate a weak version of a concept of ‘smart contracts’.”

Bitcoin coins can be locked by small scripts, not just signatures — that's how multisig ("2 of 3 keys must sign") works. So Bitcoin almost has programmable money. The paper then lists, with precision, the four walls the scripting language hits — this list is Ethereum's design spec in negative:

  • Lack of Turing-completeness — no loops (banned to prevent infinite loops during validation), so anything repetitive must be manually unrolled hundreds of times.
  • Value-blindness — a script can't say "send $1,000 worth and return the change"; coins are all-or-nothing lumps.
  • Lack of state — a coin is spent or unspent, nothing in between; multi-stage contracts (options, escrow with phases, withdrawal limits) are impossible.
  • Blockchain-blindness — scripts can't read block data like the previous hash, giving up a source of randomness and expressiveness.

Fix all four at once — loops, value-awareness, persistent state, blockchain-awareness — and you don't get a better Bitcoin; you get a general-purpose computer whose output the whole network agrees on. That's Part 2.

Part 2 · The proposal itself

Ethereum

“…what is essentially the ultimate abstract foundational layer: a blockchain with a built-in Turing-complete programming language.”

The pitch in one move: stop building blockchains for applications, build one blockchain that runs arbitrary programs, and let applications be programs. A bare-bones Namecoin becomes two lines of code (§3.3); a currency, under twenty. "Smart contracts" — the paper's definition is disarmingly concrete: cryptographic boxes that contain value and only unlock it if certain conditions are met — get the four powers Bitcoin scripting lacked: Turing-completeness, value-awareness, blockchain-awareness, and state.

2.1Ethereum Accounts

“In Ethereum, the state is made up of objects called ‘accounts’ … and state transitions being direct transfers of value and information between accounts.”

Ethereum swaps Bitcoin's coin-based state for a familiar one: named accounts with balances — the banking model from §1.1, deliberately. Every account has four fields: a nonce (transaction counter, so the same transaction can't be replayed twice), an ether balance, optional code, and a key/value storage. The crucial split: externally owned accounts (a keypair, no code — you) and contract accounts (code, no owner — a program that lives on the chain). A contract isn't a document two people signed; it's an autonomous account: every time a message arrives, its code runs, and the code alone decides what the account does — read/write its storage, send ether, message other contracts, even create new ones.

Bitcoin's state — a pile of coins (UTXO)
coin: 6 BTC → key A coin: 4 BTC → key A coin: 2 BTC → key B …spend = destroy coins, mint new ones
Ethereum's state — a map of accounts
alice · nonce 7 · 30 ETH registry · 2 ETH · code · storage{…} …transfer = edit two entries
Key idea: the "first-class citizen" property — contracts have the same powers as people. A DAO (a contract) can hire an escrow service (a contract) that pays through a savings wallet (a contract), and none of them needs to know or care which counterparties are humans.

2.2Messages, Transactions — and Gas

“If transaction execution ‘runs out of gas’, all state changes revert — except for the payment of the fees.”

A transaction is a signed package from a human-owned account: recipient, ether amount, an optional data payload — and two fields that solve the deepest problem in the paper. If anyone can upload arbitrary code with loops, what stops someone submitting an infinite loop and freezing every node that tries to validate it? (Computer science says you cannot detect this in advance — the halting problem.)

The answer is economic, not technical: gas. Every computational step costs gas. Each transaction declares STARTGAS — the most steps it's allowed — and GASPRICE — the ether it pays per step. Run out mid-execution and everything the code did is rolled back, but the fee is still paid in full. Infinite loops don't crash the network; they just bill their author until the meter cuts them off. Computation on Ethereum isn't free precisely so that it can be safe.

Anatomy of a transaction
tosignature (from) value (ETH)data […] STARTGAS — step budget GASPRICE — fee per step

2.3Ethereum State Transition Function

“The Ethereum state transition function, APPLY(S,TX) → S′, can be defined as follows…”

Here the paper cashes the §1.1 abstraction: the exact six-step recipe every node runs for every transaction. Validate signature and nonce → charge the maximum fee up front (STARTGAS × GASPRICE) → meter off gas per byte of transaction → transfer the value; if the recipient is a contract, run its code until it finishes or the gas runs out → on failure, revert everything except the fee → on success, refund unused gas and pay the miner for gas consumed. Charging up front and refunding later is the trick that makes "code that pays for itself" incentive-proof.

The paper walks one worked example through these steps — a transaction carrying 10 ether and the data [2, 'CHARLIE'] to a storage contract. The lab below reproduces that exact example, live, and lets you break it in every instructive way.

2.4Code Execution — the EVM

“…its full computational state can be defined by the tuple (block_state, transaction, message, code, memory, stack, pc, gas).”

Contract code compiles to EVM bytecode — one byte per operation, executed by a machine so simple the paper claims a basic implementation fits in a few hundred lines. (Below is one in fifteen.) The machine has three places to put data: a stack (push/pop 32-byte values — where arithmetic happens), memory (scratch space, wiped after each run), and storage (the contract's permanent key/value database — the only thing that survives). Each instruction has a defined effect on the machine state and a gas cost: ADD costs 1, but SSTORE — writing permanent state every node on Earth must keep forever — costs ~200. The fee schedule is an economics lesson disguised as a spec.

code · a working EVM in fifteen lines
// A miniature EVM: bytecode in — (status, storage, gas) out.
// Real opcode numbers: STOP=0x00 ADD=0x01 MUL=0x02 SSTORE=0x55 JUMPI=0x57 PUSH1=0x60
function run(code, gas){
  const stack = [], storage = {};
  let pc = 0;
  while (pc < code.length){
    const op = code[pc];
    const cost = op === 0x55 ? 200 : 1;   // paper: SSTORE ~200, others ~1 —
    if (gas < cost)                       // permanent state is what's expensive
      return { status: 'OUT OF GAS — all changes reverted', storage: {}, gas: 0 };
    gas -= cost;
    if (op === 0x00) break;                                        // STOP
    else if (op === 0x01) stack.push(stack.pop() + stack.pop());  // ADD
    else if (op === 0x02) stack.push(stack.pop() * stack.pop());  // MUL
    else if (op === 0x55) storage[stack.pop()] = stack.pop();     // SSTORE key,value
    else if (op === 0x57){ const d = stack.pop(); if (stack.pop()){ pc = d; continue; } } // JUMPI
    else if (op === 0x60) stack.push(code[++pc]);                 // PUSH1: next byte → stack
    pc++;
  }
  return { status: 'OK', storage, gas };
}

// compute 3 + 4·2 and store it permanently at key 0:
const program = [0x60,4, 0x60,2, 0x02, 0x60,3, 0x01, 0x60,0, 0x55, 0x00];
print('program:  ' + JSON.stringify(run(program, 250)));

// an infinite loop can't hang the network — it just runs its budget dry:
const loop = [0x5b, 0x60,1, 0x60,0, 0x57];
print('inf.loop: ' + JSON.stringify(run(loop, 100)));

2.5Blockchain and Mining

“Unlike Bitcoin, Ethereum blocks contain a copy of both the transaction list and the most recent state.”

Ethereum's block validation is Bitcoin's plus one radical addition: every block commits to a snapshot of the entire world state — every balance, every contract's storage. Validators replay the block's transactions from the previous state and check they land exactly on the state root the miner claimed. Sounds insanely wasteful — copy the whole world every ~15 seconds? The rescue is the data structure: state lives in a Merkle Patricia tree, and a block only stores the handful of tree nodes that changed; everything untouched is a pointer into the previous block's tree. One transaction touching two accounts rewrites one thin path to the root; millions of other accounts are shared, not copied.

Part 3 · What you'd build with it

Applications

“In general, there are three types of applications on top of Ethereum.”

The paper's taxonomy: purely financial applications (currencies, derivatives, savings wallets, wills), semi-financial ones (money plus a real-world side — bounties for solving hard problems), and non-financial ones (voting, governance). The section's quiet argument is that each example needs only a page or a paragraph of code — the platform, not the app, is doing the heavy lifting.

3.1Token Systems

“…all a currency, or token system, fundamentally is, is a database with one operation: subtract X units from A and give X units to B.”

Strip any currency to its essentials and it's one database operation with two conditions (sender has enough; sender approved it). That's four lines of contract code — a company's shares, a gold-backed token, loyalty points, all the same four lines with different names. On Bitcoin this took the fragile colored-coins scaffolding of §1.4; on Ethereum it's a trivial special case of "upload your own APPLY".

code · the paper's token contract, ported from Serpent
// "A currency is a database with one operation." Here is the database,
// and here is the operation — the paper's Serpent snippet in JavaScript.
const storage = { alice: 1000, bob: 0 };     // token balances (contract storage)

function tokenContract(msg){                 // runs on every incoming message
  const [to, value] = msg.data;
  if ((storage[msg.sender] || 0) >= value){  // condition: sender has enough
    storage[msg.sender] -= value;            // subtract X units from A…
    storage[to] = (storage[to] || 0) + value;// …and give X units to B
    return 'ok';
  }
  return 'rejected';
}

print('alice → bob, 250:    ' + tokenContract({ sender:'alice', data:['bob', 250] }));
print('bob → carol, 9999:   ' + tokenContract({ sender:'bob',   data:['carol', 9999] }));
print('final balances:      ' + JSON.stringify(storage));
// This paragraph of code, standardized in 2015 as ERC-20, became
// the template for essentially every token issued since.

3.2Financial Derivatives and Stable-Value Currencies

“…instead of a single issuer providing the funds to back up an asset, a decentralized market of speculators … plays that role.

Crypto's volatility problem, solved with contracts instead of banks: A and B each deposit 1000 ether into a hedging contract; it records the dollar value; 30 days later it pays A exactly the dollars' worth and B the remainder. A gets a stable asset, B gets leveraged exposure, and neither can default because the contract already holds the money in escrow. The honest caveat the paper flags itself: the contract must learn the ETH/USD price from somewhere — a data feed ("oracle") run by someone, a re-imported speck of trust in a trustless system. That speck is still the central open problem of DeFi.

3.3Identity and Reputation Systems

“The contract is very simple; all it is is a database inside the Ethereum network that can be added to, but not modified or removed from.”

The promised two-line Namecoin. First writer to a key owns it forever — that single property (public, append-only, first-come-first-served) is a name system, and it took years of independent blockchain engineering to build as a standalone coin.

code · Namecoin in two lines, as promised
// The paper's complete name-registration contract:
const storage = {};
function registry(tx){
  if (!storage[tx.data[0]]) storage[tx.data[0]] = tx.data[1];  // first writer wins
}

registry({ data: ['george', '184.72.104.150'] });   // george registers his server
registry({ data: ['george', 'evil.example.com'] }); // an impostor tries to overwrite —
print(JSON.stringify(storage));                     // — and is silently ignored, forever

3.4Decentralized File Storage

“If a contract is still paying out money, that provides a cryptographic proof that someone out there is still storing the file.”

The "decentralized Dropbox": split your file into encrypted chunks, build a Merkle tree over them, and write a contract that periodically picks a random chunk (using the previous block hash as its dice — note: blockchain-awareness from §1.5 earning its keep) and pays a small reward to whoever proves they still hold it, via exactly the Merkle-branch proofs of §1.3. Storage providers earn rent; the owner gets a continuously-audited backup. Every mechanism in this design is a §1 concept, recombined — which is the paper's real argument for generality.

3.5Decentralized Autonomous Organizations

“…a virtual entity that has a certain set of members or shareholders which, perhaps with a 67% majority, have the right to spend the entity's funds and modify its code.”

The maximal application: an organization as a contract. Members vote (say, two-thirds majority) on proposals to spend funds or change the rules; the contract counts votes and executes — bylaws with no lawyers, a treasury with no treasurer. Code being immutable, "changing the rules" works by indirection: the bylaws live at addresses stored in modifiable storage, so a passed vote swaps which code gets called. The paper distinguishes a DO (humans deliberate, then the chain enforces) from a true DAO (governance itself is automatic) and offers a lovely test: could the organization still function if no two members spoke the same language? It also — presciently — spends its time on governance mechanics rather than security. §Beyond covers how that went in 2016.

3.6Further Applications

Rapid-fire, each a paragraph in the paper: savings wallets (withdrawal limits + a recovery co-signer — the §1.5 "impossible on Bitcoin" example, now trivial); crop insurance (a derivative on a weather feed); SchellingCoin — a decentralized oracle where everyone reports a value and those inside the middle 50% get paid, betting that the only answer strangers can coordinate on is the truth; granular multisig (4-of-5 spend anything, 2-of-5 up to 0.5%/day, signatures submitted asynchronously); verifiable cloud computing with spot-checks; peer-to-peer gambling on block hashes; and prediction markets, pointing at futarchy — governance by betting.

Part 4 · The hard engineering honesty

Miscellanea and Concerns

Part 4 is the paper auditing its own design — the section that separates it from a pitch deck. Each subsection names a way the system could fail and shows the mitigation, usually economic rather than cryptographic.

4.1Modified GHOST Implementation

“…not just the parent and further ancestors of a block, but also the stale children of the block's ancestors (in Ethereum jargon, ‘uncles’) are added to the calculation of which block has the largest total proof of work backing it.”

Ethereum wants blocks every ~15 seconds, not every 10 minutes — but fast blocks break the longest-chain rule. When blocks are found faster than they propagate, honest miners constantly produce stale blocks (valid, but orphaned because someone else's block arrived first). Two poisons follow: honest work is wasted (an attacker's private chain races only against the honest blocks that survive), and big pools — who hear about their own blocks instantly — waste less, quietly compounding into centralization. The fix, adapted from Sompolinsky & Zohar's GHOST: count stale blocks ("uncles") toward a chain's total weight, and — Ethereum's own addition — pay them: an uncle earns 93.75% of a block reward, its includer 6.25%, so small miners' near-misses still count and still pay.

4.2Fees

“…the vast majority of the cost of transaction processing is borne by third parties and not the miner that is making the decision of whether or not to include it.”

A subtle market failure: the miner who includes your transaction collects the fee, but every node on the network pays to verify and store it forever — a textbook tragedy of the commons. The paper works the economics: under idealized assumptions (miner has 1/N of hashpower, all nodes equal), a miner includes a transaction when its per-step fee R exceeds NC — the cost to the whole network — so the market magically internalizes the externality. Then it lists exactly why the assumptions fail in reality (miners bear extra stale-risk, non-mining nodes verify for free, hashpower is unequal, and some attackers want to burn the commons) and installs a blunt guardrail: a floating per-block gas cap, an exponential moving average that transaction load cannot outrun.

4.3Computation and Turing-Completeness

“Turing-completeness is surprisingly easy to manage, and the lack of Turing-completeness is equally surprisingly difficult to manage.”

The counterintuitive conclusion of the whole gas design. You might expect banning loops (Bitcoin's choice) to be the safe option. But the paper shows sabotage doesn't need loops: fifty contracts that each call the next one twice explode into 2⁵⁰ computational steps with no loop anywhere. To stop that you'd need call-depth limits and step accounting — at which point you've rebuilt gas metering anyway, just without the expressiveness. Since every defense Turing-incompleteness needs is a defense Turing-completeness already has, you might as well have the loops. Meanwhile gas neutralizes the classic attacks one by one: the infinite loop bills its author; the "withdraw but run out of gas before the balance updates" trick fails because partial execution reverts; a hijacked data feed spinning forever is contained because callers can cap the gas of any sub-call.

4.4Currency and Issuance

“The permanent linear supply growth model reduces the risk of what some see as excessive wealth concentration in Bitcoin.”

Ether exists for two jobs: a common unit of exchange between on-chain assets, and — primarily — the thing gas is paid in. Denominations are pre-named to end the argument early: wei (1), szabo (10¹²), finney (10¹⁵), ether (10¹⁸) — digital-cash pioneers as SI prefixes. Issuance: a crowdsale (1337–2000 ETH per BTC — yes, 1337), 9.9% each to early contributors and a long-term endowment, then 0.26× the sale, newly mined, every year, forever. Unlike Bitcoin's hard cap, supply grows linearly — but the growth rate still falls toward zero, and the paper argues lost keys (a roughly constant fraction of supply vanishing per year) make even the absolute supply converge: at a 1% loss rate, equilibrium is 26× the sale, where annual issuance exactly replaces annual loss.

4.5Mining Centralization

“…a mining algorithm based on randomly generating a unique hash function for every 1000 nonces … to remove the benefit of specialized hardware.”

Bitcoin mining was supposed to be one-CPU-one-vote; by 2014 it was ASIC farms (custom chips thousands of times faster than any computer you own) coordinated by pools — two of which together neared 50% of the network. Ethereum's counter-plan: make the mining puzzle so computationally varied that no fixed circuit can specialize in it, and require miners to hold the full blockchain (killing the pool model where workers mine blind). The honest math: ordinary users mine on hardware they already own — marginal cost, electricity only — while ASIC builders pay for electricity and hardware. Keep specialization's edge below that (E+H)/E ratio and home mining stays viable. (The Beyond section scores how this prediction aged.)

4.6Scalability

“Like Bitcoin, Ethereum suffers from the flaw that every transaction needs to be processed by every node in the network.”

Stated without flinching: this design does not scale. Visa-level traffic would add terabytes a year, and the danger isn't disk space, it's who's left — if only a few big businesses can afford full nodes, they could collude to cheat, and light clients wouldn't notice for hours. The mitigations: mining requires full state (a floor on the number of full validators), and — the deep one — per-transaction state roots enable fraud proofs: one honest node can prove to every light client exactly which transaction broke the rules by shipping just the Patricia-tree branch needed to recheck it, with challenge-response games for data that miners refuse to publish. Trust one honest node instead of a majority of node operators. This paragraph is the acorn of the entire rollup era.

Part 5 · The user-facing vision

Putting It All Together: Decentralized Applications

“…essentially a command line application run on a virtual machine that is executed by consensus across the entire network.”

What contracts really give you, the paper admits, is a command-line program whose hard drive is the world state — powerful and unusable. A ĐApp is the missing half: a web-style interface in a special browser that talks to the chain through a JavaScript API, with business logic on Ethereum plus sibling decentralized protocols (peer-to-peer messaging, file storage) where the chain isn't the right tool. From the traditional web's point of view these pages are "entirely static": no server handles your requests — the blockchain does. This paragraph is the origin of the entire wallet-plus-web3.js-plus-frontend architecture of the modern dapp.

Part 6 · Thirty-four pages later

Conclusion

“…rather than being a closed-ended, single-purpose protocol … Ethereum is open-ended by design.”

Retrace the argument. Bitcoin proved decentralized consensus works, but hard-coded a single application into it (Part 1). Its ledger is secretly a state transition machine, so generalize the machine: accounts instead of coins, a Turing-complete VM instead of a lock script, gas to make arbitrary code economically safe, uncles to make fast blocks fair, state trees to make world-snapshots cheap (Part 2, Part 4). Then every application — money, names, insurance, organizations — collapses into a few lines of uploaded code (Part 3), delivered through interfaces that need no servers (Part 5).

The conclusion's own summary is characteristically blunt: Ethereum doesn't "support" any application — it supports a language, and applications are sentences. The bet is that an open-ended protocol — a foundational layer whose best uses hadn't been imagined yet — would matter more than any list of features. The genesis block followed in July 2015, and the platform's history since has been exactly that: things not on the paper's list.

Supplementary — not in the 2014 paper

Beyond the Whitepaper

What the following decade did to these 36 pages — confirmations, reversals, and one very expensive stress test.

2016 · code met lawThe DAO Hack & the Fork

§3.5 built to scale: "The DAO" raised ~$150M, then an attacker drained a third of it through a re-entrancy bug — the contract sent ether before updating its balance, and the recipient called back in a loop. Legal under the code's rules, theft by any human reading. Ethereum hard-forked to undo it; dissenters kept the original chain as Ethereum Classic. "Code is law" met its limits eleven months after launch.

2022 · the mergeProof of Stake

§4.5's ASIC-resistant mining held for years (GPUs, not ASICs, dominated), but the endgame was more radical: abandon mining entirely. Since 2022, validators stake 32 ETH as collateral; provable cheating gets the stake destroyed ("slashing"). Security via capital-at-risk instead of energy-burned — cutting the network's power draw ~99.95% and replacing this paper's entire mining section.

§3.1, industrializedERC-20, NFTs & DeFi

The four-line token contract became ERC-20 (2015), the standard behind the 2017 boom and today's stablecoins — §3.2's dream, now moving trillions yearly. Its non-fungible sibling ERC-721 realized "smart property" as NFTs. Lending, exchanges, and derivatives composed into DeFi — §3's list, live — along with the oracle problem §3.2 predicted, now an industry (Chainlink) rather than a footnote.

§4.6's acorn, grownRollups & Layer 2

The scalability section's fraud-proof sketch became the roadmap. Rollups execute thousands of transactions off-chain and post compressed data plus a proof to mainnet — optimistic rollups use §4.6's challenge games almost verbatim; ZK-rollups replace them with validity proofs the paper couldn't yet imagine. EIP-1559 (2021) also rebuilt §4.2's fee market: a protocol-set base fee, burned rather than paid to validators — attacking the commons problem head-on.

the unsolved sectionState Growth

§2.5's "only store the diff" elegance has a flip side the paper underplayed: state never shrinks. Every SSTORE lives forever, paid once, stored by everyone eternally — hundreds of gigabytes and counting. "State rent" and "state expiry" proposals have circulated for a decade without shipping; statelessness and Verkle trees are the current hope. The one Part-4 concern that still lacks its fix.

what survived contact with realityThe Scorecard

Held up: accounts, gas, the EVM, and "open-ended by design" — DeFi, NFTs, and DAOs were all built by strangers, permissionlessly, exactly as promised. Drifted: Serpent gave way to Solidity; ĐApp browsers became wallet extensions; ASIC-resistance ended in proof of stake; and immutable contracts holding real money made a bug a new category of catastrophe. The paper solved programmable consensus; writing correct programs turned out to be the harder problem.
